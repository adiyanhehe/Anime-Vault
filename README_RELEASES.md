# Automated AnimeVault builds & releases

AnimeVault builds installers automatically whenever `main` changes, whenever a
pull request targets `main`, whenever the workflow is run manually, or whenever
an external **release** signal is sent to the repository (via
`repository_dispatch`).

The in-app download page links can keep using GitHub's latest-release download
URL format:

```text
https://github.com/adiyanhehe/Anime-Vault/releases/latest/download/<asset-name>
```

## Triggers (the "signal")

`.github/workflows/electron-build.yml` runs on any of:

| Trigger                 | What it does                                                   |
| ----------------------- | -------------------------------------------------------------- |
| `push` to `main`        | Builds installers, creates/updates the `vX.Y.Z` tag and Release |
| `pull_request` to `main`| Builds installers and uploads them as PR artifacts (no release)|
| `workflow_dispatch`     | Manual run from the Actions tab                                |
| `repository_dispatch`   | External signal — fire from anywhere via the GitHub API        |

External signal example (curl):

```bash
curl -X POST \
  -H "Accept: application/vnd.github+json" \
  -H "Authorization: Bearer <PAT with repo:write>" \
  https://api.github.com/repos/adiyanhehe/Anime-Vault/dispatches \
  -d '{"event_type":"release"}'
```

## How the automation works

1. A commit is pushed to `main`, a PR targets `main`, the workflow is run
   manually, or a `release` event is dispatched to the repository.
2. GitHub Actions starts **Automated Installer Builds & Releases**.
3. The `desktop` job builds Windows, macOS, and Linux installers in parallel.
4. The `android` job builds the Capacitor release APK and normalises it to
   `release/AnimeVault-Android.apk`.
5. PR runs stop here — the installers are uploaded as Actions artifacts so the
   build is testable before merge, but no public Release is published.
6. On pushes to `main`, manual runs, and dispatch events, the `publish-release`
   job downloads the artifacts, reads `package.json`, force-moves the
   `vX.Y.Z` tag to the successful commit, creates/updates the matching GitHub
   Release, and uploads every installer with `gh release upload --clobber`.
7. The download page points at `/releases/latest/download/...`, so buttons
   automatically serve the newest successful release assets.

## Stable download assets

The download page expects these release asset names:

- `AnimeVault-Windows-x64.exe`
- `AnimeVault-macOS-arm64.dmg`
- `AnimeVault-Linux-x64.AppImage`
- `AnimeVault-Android.apk`

The workflow also uploads the extra artifacts generated by `electron-builder`,
including arm64 Windows installers, Intel macOS packages, Linux DEB files,
blockmaps, and update metadata files.

## What gets built

- **Windows** — NSIS installers for x64 and arm64.
- **macOS** — unsigned DMG and ZIP builds for Intel and Apple Silicon.
- **Linux** — AppImage and DEB packages.
- **Android** — Capacitor release APK.

## Desktop auto updates

The Electron app checks GitHub Releases when it launches. Release publishing
includes the `electron-builder` metadata files (`latest.yml`, `latest-mac.yml`,
and `latest-linux.yml`) so desktop auto-update delivery works out of the box.

## macOS signing note

This repository intentionally builds unsigned macOS artifacts when Apple
Developer ID credentials are not available. That avoids requiring an Apple
Developer account in CI, but it does not defeat or bypass Gatekeeper. Users may
still need to approve the unsigned app in macOS Privacy & Security settings.
For production distribution, add Apple signing and notarization secrets
instead of relying on unsigned builds.

## Local verification

To check the workflow file locally:

```bash
# 1. Make sure the YAML is well-formed
node -e "require('js-yaml').load(require('fs').readFileSync('.github/workflows/electron-build.yml','utf8'))"

# 2. Trigger a manual build (requires the `gh` CLI authenticated against the repo)
gh workflow run electron-build.yml --ref main
```
