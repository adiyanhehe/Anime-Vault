# Automated AnimeVault releases

AnimeVault now builds installers automatically whenever `main` changes, whenever a version tag such as `v0.2.0` is pushed, and when the workflow is run manually.

## What gets built

- Windows: NSIS installers for x64 and arm64.
- macOS: unsigned DMG and ZIP builds for Intel and Apple Silicon.
- Linux: AppImage and DEB packages.
- Android: Capacitor release APK.

## Desktop auto updates

The Electron app checks GitHub Releases when it launches. If a newer release exists, it downloads that version once in the background, shows progress in the in-app update cockpit, and installs it when the user clicks **Restart & install** or when the app quits after the download has completed.

Auto updates require the release assets generated by `electron-builder` plus their latest metadata files (`latest.yml`, `latest-mac.yml`, and `latest-linux.yml`) to be published to GitHub Releases. The workflow uploads every generated installer, blockmap, and updater metadata file, then creates a GitHub Release from those artifacts. Pushes to `main` get an `autobuild-<run>-<sha>` release tag automatically; pushed `v*` tags keep their real version tag.

## macOS signing note

This repository intentionally builds unsigned macOS artifacts when Apple Developer ID credentials are not available. That avoids requiring an Apple Developer account in CI, but it does not defeat or bypass Gatekeeper. Users may still need to approve the unsigned app in macOS Privacy & Security settings. For production distribution, add Apple signing and notarization secrets instead of relying on unsigned builds.

## Download page automation

The `/download` page now asks the GitHub Releases API for the newest release at runtime, matches the uploaded installer assets by platform, highlights the visitor's detected platform, and falls back to versioned links if GitHub cannot be reached. Keep release asset names recognizable (`.exe`, `.dmg`, `.AppImage`/`.deb`, `.apk`) so the page can map them automatically.


## Release job reliability

The workflow no longer asks each platform build to publish directly to GitHub Releases. Each platform only builds and uploads artifacts first. A final `release` job downloads all generated artifacts and creates or updates one release, which avoids matrix jobs racing each other or failing to create duplicate releases.
