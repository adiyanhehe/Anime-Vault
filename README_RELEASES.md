# Automated AnimeVault releases

AnimeVault builds installers automatically whenever `main` changes and whenever the workflow is run manually. The workflow creates or updates the version tag from `package.json` (for example, version `0.2.3` becomes `v0.2.3`), creates or updates that GitHub Release, and uploads the newest installer assets with stable filenames.

This means the in-app/download-page links can keep using GitHub's latest-release download URL format:

```text
https://github.com/adiyanhehe/Anime-Vault/releases/latest/download/<asset-name>
```

## How the automation works

1. A commit is pushed to `main`, or someone clicks **Run workflow** in GitHub Actions.
2. GitHub Actions starts **Automated Installer Releases**.
3. Desktop jobs build Windows, macOS, and Linux installers.
4. The Android job builds and normalizes `AnimeVault-Android.apk`.
5. The publish job reads `package.json`, creates/updates the matching `vX.Y.Z` tag at the successful commit, creates/updates the GitHub Release, and uploads all installer assets with `gh release upload --clobber`.
6. The public download page points at `/releases/latest/download/...`, so buttons automatically download the newest successful release assets.

## Stable download assets

The download page expects these release asset names:

- `AnimeVault-Windows-x64.exe`
- `AnimeVault-macOS-arm64.dmg`
- `AnimeVault-Linux-x64.AppImage`
- `AnimeVault-Android.apk`

The workflow also uploads the extra artifacts generated by `electron-builder`, including arm64 Windows installers, Intel macOS packages, Linux DEB files, blockmaps, and update metadata files.

## What gets built

- Windows: NSIS installers for x64 and arm64.
- macOS: unsigned DMG and ZIP builds for Intel and Apple Silicon.
- Linux: AppImage and DEB packages.
- Android: Capacitor release APK.

## Desktop auto updates

The Electron app checks GitHub Releases when it launches. Release publishing includes the `electron-builder` metadata files (`latest.yml`, `latest-mac.yml`, and `latest-linux.yml`) so desktop auto-update delivery can use the newest published release assets.

## macOS signing note

This repository intentionally builds unsigned macOS artifacts when Apple Developer ID credentials are not available. That avoids requiring an Apple Developer account in CI, but it does not defeat or bypass Gatekeeper. Users may still need to approve the unsigned app in macOS Privacy & Security settings. For production distribution, add Apple signing and notarization secrets instead of relying on unsigned builds.
